# 객체지향 프로그래밍
## 객체지향의 3대 속성 (3 요소)
1. 캡슐화
    * 데이터와 그 데이터를 처리하는 함수를 하나로 묶고,<br>외부에서 데이터에 직접 접근하는 것을 제한하는 것
    * 내부 구현이 어떻게 되어 있는지 외부에서 알 필요가 없게 만드는 <b>정보 은닉</b>의 성격이 강함
    * 주로 private, protected와 같은 접근 제어자를 통해 실현됨
    * 사용 이유 (의도) :
        * 데이터 보호 : 외부에서 실수로 객체 내부 데이터를 오염시키는 것을 방지하기 위함
        * 유지보수성 : 객체 내부 로직을 수정하더라도, 외부에 영향을 주지 않아 코드 수정에 용이함.
2. 상속
    * 이미 정의된 상위 클래스의 모든 특성을 하위 클래스가 물려받는 것
    * 단순히 코드를 복사하는 것이 아니라, 클래스 간 <b>계층 구조(Is-A)관계</b>를 형성
    * 사용 이유 (의도) :
        * 코드 재사용 : 공통된 기능을 부모 클래스에 한 번만 작성하면 모든 자식 클래스에서 사용할 수 있음
        * 생산성 향상 : 새로운 기능을 만들 때, 처음부터 다 만드는 것이 아니라 기존 클래스를 확장해서 빠르게 개발할 수 있다.
3. 다형성
    * 여러가지 형태를 가질 수 있다는 의미로 <br>
    하나의 인터페이스나 추상 클래스를 통해 여러 타입의 객체를 다룰 수 있는 것
    * 같은 이름의 메서드를 호출하더라도, 각 객체가 자신의 방식대로 다르게 동작하는 것
        * 오버라이딩과 오버로딩이 대표적인 예시임
    * 사용 이유 (의도)
        * 유연성과 확장성 : 새로운 클래스가 추가되더라도 기존 코드의 수정이 필요하지 않음.<br>
        상위 클래스의 함수를 호출하여 하위 클래스에서 작성된 구체적인 동작을 실행할 수 있기 때문.
        * 결합도 감소 : 구체적인 클래스 타입이 아니라 추상적인 인터페이스에 의존하게 만들어
        코드 간 의존성을 줄여줌.

### 오버라이딩 (override, 재정의) vs 오버로딩 (overload, 다중적재)
* 오버라이딩 (override) : 상속관계에서 상위 클래스의 메서드를 하위 클래스에서 재정의하는 것
    * 상위 클래스의 기능을 아예 재정의하거나 상위 클래스의 기능을 확장하기 위함.
* 오버로딩 (overload) : 하나의 메서드를 매개변수의 구성을 다르게해서 호출할 수 있도록 하는 것
    * 서로 다른 매개변수의 구성으로도 하나의 기능을 호출하기 위함.

## SOLID 원칙
* 객체지향 설계에서 SW의 유지보수성, 재사용성, 확장성을 향상시키기 위한 5가지 규칙
* 각 원칙들은 서로 보완적인 특성이 있다.
### SRP : 단일 책임 원칙
* 클래스는 오직 하나의 책임을 가지고 있어야 한다.
* 클래스가 수정될 이유는 단 하나의 책임에 대한 것이어야 한다.
### OCP : 개방-폐쇄 원칙
* 클래스는 수정엔 닫혀있고 확장엔 열려 있어야 한다.
* 새로운 기능을 추가할 땐 클래스의 수정보단 확장하는 방식으로 진행해야한다.
### LSP : 리스코프 치환의 원칙
* 하위 클래스는 상위 클래스의 위치를 대체(치환)할 수 있어야 한다.
* 상위 클래스의 기능을 하위 클래스도 수행할 수 있어야 함을 의미.
### ISP : 인터페이스 분리 원칙
* 클래스는 불필요한 인터페이스를 구현하지 않아야 한다.
* 지나치게 비대한 인터페이스보단 각 기능별 작은 인터페이스로 나누어 필요한 기능만 구현해야 함
### DIP : 의존 역전 원칙
* 고수준 모듈은 저수준 모듈에 의존해선 안된다.
* 추상화는 세부 사항에 의존해선 안되며, 세수 사항이 추상화에 의존해야한다.
