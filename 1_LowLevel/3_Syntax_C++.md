# C++ 문법
## C++ 빌드 과정
1. 전처리 `Preprocessing`
    * 이 단계에서는 소스 파일(.cpp) 등이 전처리기에 의해 처리됩니다.
    * 전처리기 지시문(#include, #define, #if 등)이 실행되어 코드에 포함시킵니다.
        *  예를 들어 #include 지시문은 해당 헤더 파일의 내용을 소스 파일에 복사합니다.
    * 이 단계에서의 출력은 전처리된 소스 코드이며 사람은 읽기 어려운 형태입니다.
2. 컴파일 `Compilation`
    * 전처리된 소스 코드는 컴파일러에 의해 기계어로 변역됩니다.
    * 컴파일러는 각 소스 파일(.cpp)을 오브젝트 파일(.obj 또는 .o)로 변환합니다.
    * 이 단계에서는 구문 분석, 의미 분석, 코드 최적화 등이 수행됩니다.
3. 어셈블리 `Assembly`
    * 컴파일 단계에서 생성된 오브젝트 파일들을 기계어 코드(어셈블리어)를 포함하고 있습니다.
    * 어셈블리 단계는 일반적으로 컴파일 단계에 내장되어 잇으며, 별도의 단계로 분리되지 않는 경우가 많습니다.
4. 링킹 `Linking`
    * 컴파일러가 생성한 오브젝트 파일들이 링커에 의해 하나의 실행 파일로 결합됩니다.
    * 링커는 각 오브젝트 파일들 간의 의존성을 해결하고, 필요한 라이브러리 코드를 포함시켜 실행 가능한 파일(.exe or .out)을 생성합니다.
    * 이 단계에서 외부 라이브러리 함수에 대한 참조, 전역변수의 메모리 할당 등이 이루어집니다.

## C++ 컴파일 과정
* C++ 컴파일 과정은 소스 코드가 기계어로 변역되는 일련의 단계를 포함합니다.
* 이 과정은 복잡한 소프트웨어 프로젝트를 실행 가능한 형태로 변환하는데 필요합니다.
1. 전처리
    * 전처리기는 소스 파일(.cpp)을 처리합니다.
    * 전처리 지시문이 실행됩니다.
        * #include 해당 헤더 파일의 내용을 소스 파일에 삽입
        * #define(매크로 정의), #if(조건부 컴파일 지시문)은 해당 조건에 따라 코드를 조정합니다.
    * 이 단계의 결과는 전처리된 코드이며 다음 단계에서 컴파일러에 의해 처리됩니다.
2. 컴파일
    * 전처리된 소스 코드는 컴파일러에 의해 기계어로 변환됩니다.
    * 컴파일러는 소스 파일을 오브젝트 파일로 변환합니다.
    * 이 단계에서 구문 분석(Syntax Analysis), 의미 분석(Semantic Analysis), 중간 코드 생성(Intermediate Code Generation), 최적화(Optimization) 등의 과정이 수행됩니다.
3. 링킹
    * 컴파일 단계에서 생성된 오브젝트 파일등은 링커에 의해 하나의 실행 파일로 결합됩니다.
    * 링커는 오브젝트 파일들 간의 의존성을 해결하고, 필요한 라이브러리 코드를 포함시켜 최종적인 실행 가능한 파일을 생성합니다.
    * 이 단계에서는 외부 라이브러리 함수에 대한 참조 해결, 전역 변수의 메모리 할당등이 이루어집니다.
* c++ 컴파일러는 이러한 각 단계를 통해 고수준의 소스 코드를 저수준의 기계어 코드로 변환하는 복잡한 작업을 수행합니다.
* 컴파일 과정의 정확한 세부 사항은 사용하는 컴파일러의 종류(예: GCC, Clang, MSVC 등)와 설정에 따라 다를 수 있습니다.

## 아스키 문자란? 현대에선 어떻게 쓰이나?
* 아스키 문자는 컴퓨터에서 텍스트를 표현하기 위해 사용되는 문자 인코딩 체계입니다.
* 1960년대 개발된 이 인코딩은 영어 알파벳, 숫자, 일부 특수 문자, 그리고 제어 문자를 나타내는데 사용됩니다.
### 특징
* 아스키 문자 집합은 총 128개의 문자로 구성되어 있으며, 각 문자는 7비트로 표현됩니다.
    * 실제 컴퓨터에선 1 byte를 사용
* 이 문자 집합에는 대문자와 소문자 영문 알파벳, 숫자, 표준 특수 문자, 그리고 제어 문자가 포함됩니다.
### 현대에서의 사용
* 아스키는 초기 컴퓨터 시스템에서 기본적인 문자 인코딩으로 널리 사용되었으며, 현재도 많은 시스템과 프로토콜에서 기본적인 텍스트 인코딩으로 사용되고 있습니다.
* 그러나 현대에는 몇 가지 중요한 이유로 인해 아스키보다 더 진보된 문자 인코딩 체계가 널리 사용됩니다.
1. 다국어 지원
    * 아스키는 주로 영어를 위해 설계되었기 때문에 다양한 언어와 문자를 지원하는데 한계가 있습니다.
    * 이를 극복하기 위해 UTF-8, 16과 같은 유니코드 인코딩 체계가 개발되었으며, 이들은 전 세계와 다양한 언어와 문자를 지원합니다.
2. 확장성
    * 아스키는 상대적으로 제한된 수의 문자만을 지원합니다.
    * 유니코드와 같은 확장된 인코딩 체계는 훨씬 더 많은 문자와 기호를 포함할 수 있어, 다양한 기술적 요구 사항을 충족시킬 수 있습니다.
3. 호환성
    * 많은 현대 시스템과 프로그래밍 언어는 유니코드를 기본 인코딩으로 사용하며, 아스키 문자 집합은 유니코드와 호환됩니다.
    * 예를 들어, UTF-8 인코딩은 아스키 문자 집합의 범위 내에서 아스키와 동일한 바이트값을 사용합니다.
* 결론적으로 아스키는 현대 컴퓨터 시스템에서 여전히 기본적인 텍스트 인코딩으로 사용되고 있으나, 다국어 지원과 확장성을 위해 유니코드 인코딩 체계가 점점 더 선호되고 있습니다.

## 포인터와 참조의 공통점, 차이점, 사용처에 대해 아는대로 설명해주실래요?
### 공통점
* 복사 비용을 줄이고 싶을 때, 간접적으로 객체에 접근해서 원격으로 원본을 건드리고 싶을 때 사용할 수 있습니다.
### 차이점
* 포인터는 NULL 값이 허용되며, 재할당이 가능해서 다른 주소를 가리킬 수 있습니다.
* 참조는 항상 유효한 객체를 가리켜야 하므로 선언 시 반드시 초기화되어야 하며, 한 번 초기화되면 다른 객체를 가리킬 수 없습니다.<br> 또한 NULL값으로 초기화될 수 없습니다.
### 사용처
* 포인터는 동적 할당, 선택적 접근, 데이터 구조 등에 주로 사용됩니다.
* 참조는 함수의 매개변수나 반환 값으로 사용할 때, 그리고 클래스 내부에서 객체 또는 데이터를 참조할 때 선호됩니다.

## 가상 함수에 대해 아는대로 설명해주실래요?
* C++에서 Virtual 키워드를 사용하는 가상 함수는 다형성을 구현하게 해줍니다.
* 이러한 가상 함수의 목적은 런타임에 함수의 바인딩을 결정하는 것입니다.
* 이를 통해서 자식 클래스에서 부모 클래스의 메서드를 오버라이딩하고, 부모 클래스의 포인터나 참조를 통해 해당 메서드를 호출할 때, 자식 클래스의 구현을 실행할 수 있습니다.
* 이 과정은 동적 바인딩이라고 합니다.
### 가상 함수 테이블
* C++ 컴파일러는 가상 함수를 하나라도 포함하는 클래스에 대해 4 byte 크기의 가상 함수 테이블을 생성합니다.
* 이 테이블은 클래스별로 하나씩 생성되며, 클래스 내의 모든 가상 함수 주소를 포함합니다.
* 이 가상 함수 테이블은 모든 인스턴스에 의해 공유되는 구조입니다.
* 객체가 생성될 때, 각 인스턴스는 가상 함수 테이블의 시작 주소를 가리키는 포인터 `vptr`를 내부에 저장합니다.<br> 이 포인터를 사용해서 해당 클래스의 가상 함수 테이블에서 어느 주소에 있는 함수가 호출되어야 하는지를 결정합니다.
* 만약 자식 클래스가 부모 클래스의 가상 함수를 오버라이드하지 않으면, 자식 클래스의 가상 테이블은 부모 클래스의 가상 함수 주소를 보관합니다.

## 모든 함수를 가상 함수로 쓰면 안되나요?
* 모든 함수들을 가상 함수로 만듦으로써, 언제나 동적 바인딩이 제대로 동작하게 만들 수 있습니다.<br>(실제로 java에선 모든 함수들이 디폴트로 virtual 함수로 선언됩니다.)
* 그럼에도 불구하고 C++에서 사용자가 직접 선언하도록 한 이유는 가상 함수를 사용하게 되면 약간의 오버헤드가 발생하기 때문입니다.
    * 함수를 바로 호출하는 것이 아니라 가상 함수 테이블을 한 단계 더 거쳐서 실제로 어떤 함수를 고를지 결정하는 과정입니다.
* 그래서 가상 함수를 호출하는 경우, 일반적인 함수보다 약간 더 시간이 오래 걸립니다.
* 물론 이 차이는 극히 미미하지만, 최적화가 매우 중요한 분야에서는 이를 감안할 필요가 있습니다.
    * C++은 다른 언어와 달리, 멤버 함수가 디폴트로 가상함수가 되지 않습니다.

## 상속 환경에서 소멸자를 어떻게 다뤄야 하는지 설명해주세요.
* 상속 환경에서 클래스를 동적할당하고 제거할 때, 자식 클래스에서도 소멸자에 virtual 키워드를 붙여주어야 안전합니다.

* 자식 클래스의 소멸자에 virtual 키워드를 붙이지 않은 경우
    * 동적할당한 인스턴스의 타입이 자식 클래스의 타입이라면 문제없이 자식 클래스 -> 부모 클래스 순으로 소멸자가 호출될 것입니다.
    * 문제는 자식 클래스를 동적할당하고 부모 클래스의 포인터로 받은 경우에 delete를 호출하면
부모 클래스의 소멸자만 호출되고, 자식 클래스의 소멸자는 호출되지 않습니다.
    * 이는 자식 클래스에서 추가한 데이터에 대한 메모리가 정리되지 않아 메모리 누수로 이어집니다.
    * 이런 상황을 해결하기 위해서 자식 클래스의 소멸자에 virtual 키워드를 붙여서,
    부모 클래스를 통한 소멸자 호출에도 자식 클래스의 소멸자가 호출되도록 할 수 있습니다.

## 연산자 오버라이드는 왜 사용하는가
* int, float 같은 원시 자료형들과 같이 커스텀 타입에 대해서도 사칙연산, 대입과 같은 연산자를 사용하기 위함.
* 즉, 커스텀 타입에 대해서도 코드의 일관성 있도록 사용하기 위해서 연산자 오버라이드를 사용.
* 추가로 비교 연산자같이 특정한 연산자들을 오버라이드함으로써 STL과 알고리즘 라이브러리에 응용해서 사용할 수 있음.

## L-Value vs R-Value
* L-Value : 다시 호출할 수 있는 메모리
    * 일반적으로 선언한 변수들
* R-Value : 한번만 호출할 수 있는 일회성 메모리
    * 일반적으로 리터럴 상수들
    * 단, 리터럴 중에서도 문자열 리터럴은 L-Value임
        * ex) `string str = L"Hello";`
* 문자열 리터럴이 L-Value인 이유
    * 문자열은 배열 형태의 데이터를 식별할 수 있는 메모리 위치를 가지므로 L-Value로 분류됨
        * 문자열 리터럴은 프로그램의 데이터 영역(주로 ROM)에 저장됨
        * 식별자(주소)가 있어서 `&"Hello"`와 같이 주소 연산자를 쓸 수 있음
    * 배열로의 붕괴(Decay)
        * `const char*` 타입으로 표현될 때, 문자열 리터럴은 `const char[N]` 타입의 L-Value에서 `const char*` 로 암시적 형변환된다.
        * 배열이기 때문에 시작 주소를 통해서 `const char*`로 암시적 형변환되는 것

## Move Semantics : 이동 의미론
* std::move() : L-Value를 R-Value로 변환해서 반환시켜준다.
* 반환시칸 R-Value를 변수가 대입받으면 기존 변수의 R-Value의 소유권은 사라진다.

### std::move 역할
* L-Value를 R-Value를 바꿔주는 역할. <br> 실제로 이동을 해주는 함수가 아님.
* 이를 통해서 이동 연산자가 정의되어 있는 경우 변경된 R-Value로 오버로딩 될 수 있음
```cpp
template<typename T>
typename remove_reference<T>::type&& move(T&& param)
{
	using ReturnType = typename remove_reference<T>::type&&;
	
	return static_cast<ReturnType>(param);
}

// C++ 14 이후
template<typename T>
decltype(auto) move(T&& param)
{
	using ReturnType = remove_reference_t<T>&&;

	return static_cast<ReturnType>(param);
}

std::vector<std::string> v;
std::string str = "example";
v.push_back(std::move(str));  // str 은 이제 껍데기만 남음
str.back();                   // 정의되지 않은 작업!
str.clear();                  // 다만 clear 자체는 가능하다.
```
* 때문에 실제로 하는 일은 move가 아니라 R-Value Cast임

### std::forward 역할
* std::forward()의 경우 move와 다르게 R-Value일 때는 R-Value로 전달, L-Value일때는 L-Value로 전달하는 역할
* 원래 c++에서는 참조에 대한 참조는 안되지만, 템플릿 인스턴스화에선 참조에 대한 참조가 허용됨.
    * `template` 메서드를 사용할 때, R-Value 레퍼런스로 매개변수로 넘기더라도 L-Value로 참조됨.
* C++에서는 연역 과정에서 둘 중 하나라도 L-Value 참조라면 결과는 L-Value 참조가 된다는 규칙이 있어 forward를 통해서 L-Value, R-Value를 구분할 수 있음.
```cpp
Some& && forward(typename remove_reference<Some&>::type& param)
{	
    return static_cast<Some& &&>(param);
}
```
* Some&& 의 경우에는 T가 Some로 연역됨

### move vs forward
* move : 이동 준비가 목적 (L-Value -> R-Value 변환)
* forward : 전달일 목적 (L-Value는 L-Value로, R-Value는 R-Value로)

## Cast
* 캐스팅 (형변환)은 명시적 형변환과 암시적(묵시적) 형변환으로 나뉨
### 암시적 형변환
* 다른 표기 없이 호환되는 타입들 간 변환할 수 있게 해줌
    * 데이터 손실이 발생할 수 있음
    * 호환되는 타입을 사전에 구현해야함.
    * 유지보수 측면에서도 알아내기 어렵다는 단점이 있음.

### 명시적 형변환
* 형변환 시, 변환할 타입을 명시해서 변환하는 것
```cpp
// C 스타일
float f = 1.1f;
int a = (int)f;
```
* C++에서는 여러 캐스팅 연산자들을 사용함
    * C 스타일보다 가독성이 높고 위험성을 줄여줌
### `static_cast<T>`
* 기존 C 스타일 형변환 대신 사용
* 형변환이 가능한지에 대한 검사는 컴파일 타임에서 이루어짐
* 단, 다운캐스팅(상속 관계에서 상위->하위 클래스)의 경우 타입 안정성을 보장하지 않음
    * 실제 객체가 변환하려는 타입이 아닌 경우에도 컴파일러는 오류를 발생시키지 않음

```cpp
float f = 1.1f;
int a = static_cast<int>(f);
```
### `dynamic_cast<T>`
* 런타임에 타입 안정성을 검사 (RTTI)
    * 주로 다형성을 가진 클래스 (가상 함수를 가진 클래스)의 포인터나 참조 간의 변환에 사용
    * 가상 테이블을 이용해서 캐스팅이 가능한지 검사
* 다운캐스팅 시, 실제 객체 타입이 목표 타입과 일치하는지 확인하고, 일치하지 않으면 nullptr을 반환
    * 이를 통해 타입 안정성을 제공
* 런타임에 타입 안정성 검사를 진행하고 상속 관계에서 업캐스트나 변환을 지원해주지만 다른 데이터형 변환은 허용하지 않는다.
* 업캐스팅만이 가능하고 만약 가상함수까지 사용한다면 다운캐스팅 또한 지원해줌.

### 강제 캐스팅을 사용하는 것보다 static_cast, dynamic_cast를 사용할 때의 이점
* 컴파일러가 오류 체크를 진행해줌
    * 강제 캐스팅을 사용하면 런타임시 seg fault, runtime error 등 예기치 못한 에러를 발생시킬 수 있는데 컴파일러단에서 오류를 잡아 추후 문제가 될 가능성을 줄여주는 이점을 가지고 있다.
* 상속관계에 있어도 형변환이 가능
    * 다운캐스팅에서 형변환은 안전하지 못하기 때문에 dynamic_cast를 이용한 방법은 안정성을 보장함

### `const_cast<T>`
* const나 volatile 를 제거, 부여하는데 사용
    * 다만 부여의 경우에는 사용하는 경우가 잘 없음
* 주로 상수 객체에 대한 수정이 필요할 때 사용
* 타입 자체 변환보다는 객체의 속성을 변경하는데 초점을 맞춤
```cpp
void NonConstFunc(Some s);

//...
const Some s;
NonConstFunc(s);   // ERROR!
NonConstFunc(const_cast<Some>(s))
```
### `reinterpret_cast<T>`
* 모든 포인터 유형을 다른 포인터 방식으로 변경할 수 있음.
    * 또한 포인터 타입을 정수로 바꾸는데도 가능한 캐스팅
* 비트 단위로 메모리를 분해, 재조립하는 강력한 형변환을 하기 때문에 이해도가 필요

### 업캐스팅과 다운캐스팅에 대해 설명해주시겠어요?
* 업캐스팅과 다운캐스팅은 상속 관계에 있는 클래스들 간의 형변환 방식입니다.
* 업캐스팅 : 자식 클래스의 객체를 부모 클래스 타입으로 변환하는 것입니다.
    * 업캐스팅은 항상 안전하며 암시적으로 수행될 수 있습니다.
    * 업캐스팅을 통해 자식 클래스의 객체가 부모 클래스의 인터페이스를 사용할 수 있게 됩니다.
* 다운캐스팅 : 부모 클래스 타입에서 특정 자식 클래스의 타입으로 변환하는 것입니다.
    * 다운캐스팅은 항상 안전하진 않고, 명시적으로 수행되어야 합니다.
    * 캐스팅하려는 객체가 실제로 자식 클래스일 때만 안전하게 수행될 수 있습니다.
    * 잘못 하용하면 정보 손실의 위험이 있어, 이를 보완하기 위해서 dynamic_cast를 사용합니다.
        * 런타임에 객체의 타입을 확인하는 (RTII) 안전한 캐스팅을 가능하게 합니다.
        * 만약 캐스팅이 불가능하다면 nullptr을 반환합니다.


## 전역 변수 vs static
* 데이터 영역에 저장된다는 공통점
* 접근 범위의 차이
    * 전역변수는 특정 클래스나 네임스페이스에 국한되지 않고 어디서든 접근 가능
    * static은 대체로 객체 내부에서 선언하여, 객체를 생성하지 않고도 접근 가능
        * 단, static의 경우 선언된 객체의 범위지정연산자(::)를 통해서 접근해야 함.
        * 이를 통해서, 선언된 객체 내 파일 범위나 블록 범위 내로 제한되어 프로젝트 내에서 이름 충돌이 방지됨

## 함수 포인터와 람다
* 함수 포인터 : 선언된 함수의 포인터(메모리 주소)를 담을 수 있는 변수
* 람다 : 일회성 함수
    * 캡쳐절을 통해서 람다식 생성 당시에 주변 데이터를 담을 수 있다.
    * 캡쳐절이 없다면 매개변수를 이용하는 등의 작업을 수행하는 번거로움이 있음

## 순수 가상 함수와 추상 클래스의 차이점
* 추상 클래스는 하나 이상의 순수 가상 함수(pure virtual function)를 포함하는 클래스
    * 추상 클래스의 경우 실행 코드가 없은 순수 가상 함수를 가지고 있어 인스턴스를 만들지 못합니다.
    * abstract 키워드를 통해서 순수 가상 함수가 없더라도 클래스를 추상 클래스로 선언할 수 있습니다.
        * 이 경우, 해당 클래스에 비록 순수 가상 함수가 없더라도 인스턴스를 만들지 못하다록 하여 반드시 상속을 통한 사용을 강제할 수 있습니다.
    * C++에선 인터페이스를 제공하지 않아, 순수 가상 함수를 인터페이스 역할로 사용합니다.
* 순수 가상 함수는 함수 본체가 정의되지 않은 가상 함수로, 상속한 하위 클래스에서 반드시 오버라이드 해야 합니다.

### 순수 가상 함수와 일반 가상 함수의 차이
* 순수 가상 함수는 반드시 하위 클래스에서 재정의 해줘야합니다.
* 일반 가상 함수는 함수의 본체가 있기 때문에 재정의가 필수가 아닙니다.

## 상등성과 동등성
* 상등성과 동등성은 객체를 비교하는 두 가지 다른 개념
1. 상등성(Identity)
    * 두 객체가 메모리 상에서 정확히 같은 위치에 있는지, 즉, 두 객체가 완전히 동일한 객체인지를 나타냄
    * 상등성 비교는 주로 포인터나 참조를 통해 이루어짐
2. 동등성(Equality)
    * 두 객체의 내용이나 상태가 동일한지를 나타냄.
    * 이는 객체의 실제 값 또는 상태를 비교하는 것으로 두 객체가 다른 메모리 위치에 있더라도,
내용이 같은면 동등하다고 할 수 있습니다.

## C++ 생성자 Constructor에 대해 아는대로 설명해주실래요?
* C++에서 생성자는 클래스의 인스턴스가 생성될 때, 자동으로 호출되는 특별한 멤버 함수
* 생성자의 주요 목적은 객체를 초기화하는 것
* 생성자의 이름은 클래스의 이름과 동일
* 생성자는 반환 타입을 가지지 않으며, 오버로드가 가능
### 주요 생성자 종류
1. 디폴트 생성자 : 아무런 인자를 받지 않는 생성자. 사용자가 명시적으로 디폴트 생성자를 정의하지 않으면, 컴파일러는 자동으로 디폴트 생성자를 생성합니다.
    * `SomeClass()`
2. 파리미터가 있는 생성자 : 하나 이상의 매개변수를 가지고 객체의 초기 상태를 설정하는데 사용합니다.
    * `SomeClass(Type P1, ...)`
3. 복사 생성자 : 다른 객체의 복사본을 만드는데 사용합니다. 상수 L-Value&를 매개변수로 사용합니다.
    * `SomeClass(const Type& CopyParam)`
4. 이동 생성자 : C++ 11에서 도입되어 다른 객체의 리소스를 '이동'해서 받을 때 사용합니다.
    * `SomeClass(Type&& MoveParam)`
    * const가 붙지 않는 이유는 이동 이후에 원본 객체의 리소스를 nullptr로 만들기 때문

## C++ 클래스에서 암시적으로 생성되는 함수가 있나요?
* 디폴트 생성자 : 객체가 생성될 때 아무런 인자를 받지 않을 때 호출
* 복사 생성자 : 객체가 다른 객체의 복사본으로 생성될 때 호출됨
* 복사 할당 연산자 : 객체에 다른 객체의 내용을 복사할 때 사용됨
    * `Type& operator=(const Type& CopyParam)`
* 이동 생성자 : 임시 객체의 리소스를 이전할 때 사용
* 이동 할당 연산자 : C++ 11 이후 추가된 기능으로, 임시 객체의 리소스를 다른 객체에 이전할 때 사용
    * `Type& operator=(Type&& MoveParam)`
* 소멸자 : 객체의 수명이 끝날 때 호출되어 리소스를 정리

## C++에서 class와 struct의 차이
* 공통점 : 멤버 함수, 생성자, 그리고 소멸자를 가질 수 있고 상속을 사용할 수 있다.
* 차이점
    * class : 기본 접근 지정자가 private (정보 은닉을 위함)
    * struct : 기본 접근 지정자가 public
    * 문법적 차이는 이외에 없음
* C++ 표준에서는 의미적 차이를 규정하지 않지만, C++ 커뮤니티와 주요 스타일 가이드는 의도를 코드로 표현하는 수단으로 구분해서 사용하는 것을 권장하고 있습니다.
    * class : 캡슐화된 능동적 객체
        * 불변식 존재
        * 생성자로 상태를 보장
        * 캡슐화, 다형성이 목적
        * 인터페이스와 구현의 분리
        * 비즈니스 로직, 서비스에 사용
    * struct : Plain Old Data으로 구성된 수동적인 데이터 묶음
        * 불변식 없음
        * 생성자가 없어도 무방
        * 단순 집계(aggregate) : 데이터의 묶음이 목적
        * 캡슐화가 불필요한 데이터
        * 알고리즘 파라미터, 설정값 등의 사용
### 판단 기준
1. 멤버변수들에 서로 독립적으로 직접 접근되어도 되는가?
    * yes -> struct 후보
2. 불변식이 존재하는가?
    * 불변식 invariant : 객체, 데이터 멤버, 혹은 제어 흐름이 항상 참이 되어야 하는 상태
    * yes -> class 사용
3. 생성자에서 상태를 검증/초기화 해야하는가?
    * yes -> class 사용
4. 다형성이나 상속 계층이 필요한가?
    * yes -> class 사용
* 2~4. 모두 no 라면 struct가 적합
