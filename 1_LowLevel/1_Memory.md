# 메모리
## 1. 원시 포인터와 스마트 포인터 차이
- 원시 포인터는 데이터의 메모리 주소를 가리키는 기능만 수행
- 스마트 포인터는 메모리 주소를 가리키는 기능과 함께 데이터의 메모리 관리 기능을 추가됨.
추가적으로 메모리에 대한 사용 권한 또는 소유 권한을 나타낼 수 있음.
    - 메모리 관리적 측면에서는 더 이상 참조가 없는 경우에 메모리를 자동 정리.
    - 메모리에 소유에 대한 권한을 처리하는 방식이 크게 3가지가 있음.
        1. `shared_ptr`
        2. `weak_ptr`
        3. `unique_ptr`

- 포인터 관리 뿐만 아니라 소유에 대한 권한에 대한 것

### 1-1. 스마트 포인터 세종류들의 차이
- 스마트 포인터 - 메모리에 대한 권한 처리 방법
1. `shared_ptr`
    - 여러 곳에서 데이터의 메모리에 대한 참조가 가능
    - reference count를 기반으로 메모리 참조 횟수를 알 수 있으며, reference count가 0이 되면 메모리를 정리해줌.
        - 정확히는 strong reference count
    - shared_ptr을 쓰는 이유 : 메모리에 대한 수명 연장
        - 공유한다는 것은 한 데이터에 대해서 여러 곳에서 소유권을 주장하는 것.
        - 여러 곳에서 하나의 데이터에 대한 메모리에 공유할 때는 댕글링 포인터에 대해 주의해야함.
        - 한 곳에서 아직 사용 중인데 다른 곳에서 메모리를 해제하는 등의 문제로 방지하기 위해서
        reference count를 이용해서 메모리에 대한 수명을 유지할 수 있음.
2. `weak_ptr`
    - shared_ptr과는 반대로 소유권을 주장하지 않고 참조하고자할 때 쓰는 것이 weak_ptr.
        - shared_ptr에 대해 strong reference count를 증가시키지 않고 참조
    - 메모리 해제에 관여하지 않고 참조할 수 있음.
        - 이 때문에 weak_ptr이 가리키는 곳은 nullptr일 수 있음.
        - 그러므로 IsValid() 를 통해서 가리키는 메모리 주소가 살아있는지 확인해야 함.
    - weak_ptr을 쓰는 이유 : 
        - shared_ptr의 사용에서 순환 참조 문제가 발생할 수 있음.
        - 서로 다른 메모리가 서로를 참조하고 있기 때문에 reference count가 1로 유지되어 해제되지 않는 문제 -> 메모리 누수 문제
        - 이 문제를 weak_ptr을 적용하여 두 메모리가 약참조를 하도록하여 순환 참조를 해결할 수 있음
        - 이와 함께, 설계적인 의도에서 사용할 수도 있는데 어떤 데이터의 소유권은 다른 곳에 있고
        weak_ptr로 참조하는 객체는 단순히 의존만한다는 의도를 명시할 수 있음
3. `unique_ptr`
    - 데이터에 대해서 단 한 곳에서만 참조 가능 -> 유일한 소유권 의미
        - 데이터의 소유 주체를 명확히 할 수 있음
    - 단 한 곳에서만 소유가 가능하기 때문에 복사 연산이 불가능
        - 다른 곳으로 소유권을 넘기고 싶다면 move sematic을 사용해서 이동시킬 수 있음
        - 또는 release()를 통해서 원시 포인터를 받고 소유권을 포기하는 것이 가능
    - unique_ptr은 선언된 스코프를 벗어나면 자동으로 메모리를 정리 (RAII)
        - 클래스에 멤버변수로 unique_ptr이 있다면, 클래스가 소멸될 때 같이 정리됨
        - 함수 내에서 선언되었다면, 함수 호출이 끝나고 스택 영역에서 정리될 때 같이 정리됨

## 2. 얕은 복사 vs 깊은 복사
- 실질적인 데이터의 복사 없이, 한 데이터의 메모리를 참조하는 것을 얕은 복사
- 새로운 메모리 주소를 할당받아 기존의 데이터 자체를 복사하는 것을 깊은 복사

## 3. L-Value vs R-Value
* L-Value : 다시 호출할 수 있는 메모리
    * 일반적으로 선언한 변수들
* R-Value : 한번만 호출할 수 있는 일회성 메모리
    * 일반적으로 리터럴 상수들
    * 단, 리터럴 중에서도 문자열 리터럴은 L-Value임
        * ex) `string str = L"Hello";`
* 문자열 리터럴이 L-Value인 이유
    * 문자열은 배열 형태의 데이터를 식별할 수 있는 메모리 위치를 가지므로 L-Value로 분류됨
        * 문자열 리터럴은 프로그램의 데이터 영역(주로 ROM)에 저장됨
        * 식별자(주소)가 있어서 `&"Hello"`와 같이 주소 연산자를 쓸 수 있음
    * 배열로의 붕괴(Decay)
        * `const char*` 타입으로 표현될 때, 문자열 리터럴은 `const char[N]` 타입의 L-Value에서 `const char*` 로 암시적 형변환된다.
        * 배열이기 때문에 시작 주소를 통해서 `const char*`로 암시적 형변환되는 것
